# Church Machine — Lambda Calculus Reference Architecture

**v1.0 — May 2026**
**CONFIDENTIAL**

---

## Overview

The Church Machine is a Lambda Calculus machine at the hardware level. This is not a metaphor or a pedagogical frame: the three primitives of the untyped lambda calculus — **variables**, **abstractions**, and **applications** — map directly onto specific hardware structures and instructions. The Boot C-List is the initial environment Γ. A Golden Token is a lambda binder. A CALL instruction is β-reduction. TPERM is a runtime type judgment.

This document traces each correspondence in detail, giving educators, students, and hardware architects a single authoritative reference for *why* the Church Machine is named as it is.

---

## 1. Primitives-to-Hardware Mapping

| Lambda Primitive | Church Machine Mechanism | Brief Correspondence |
|---|---|---|
| **Variable** `x` | C-List slot / CR register loaded by `LOAD` | A variable name is an index into the current environment Γ (CR6); `LOAD CR0, CR6, 4` resolves name 4 from Γ into CR0. |
| **Abstraction** `λx.e` | NS slot + lump binary; E-GT as binder | The lump body is the term `e`; the lump's private C-List is the closed environment; the E-GT is the handle that names the abstraction. |
| **Application** `f x` | `CALL CRn` / `ELOADCALL` instruction | CALL validates the E-GT in CRn (checks `f` is a function), switches CR6 to the callee's closed environment, and transfers control to the callee's code — exactly β-reduction. |

## 2. The Environment Model (Γ) — Boot C-List

In lambda calculus, the environment Γ maps variable names to their values. On the Church Machine, the environment is an **Abstraction** in a **Namespace**: directional hierarchy array of Golden Tokens, the DNA structure of a functional software application structures the fault-tolerant configuration using one per slot of meta-data per Golden Token, indexed by a word offset in the immutable Golden Token.

At boot the hardware loads the predefined **'Boot' Namespace** into CR15. This the resident object list of Golden Token Meta Data slots. This array is the list Γ for every function abstraction that runs from boot. This Namepace list is the complete set of static, named digital objects called **Lumps,** each one referenced by Golden Tokens owned by individual Abstractions in their own private C-List of GTs each one with explicite usage permissions. The boot abstraction can see only the GTs in its own private c-list following the **POLA** the principle of least authority. No other objects exist at this point.

### DEMO_CLIST — 18-Slot Boot Environment

| Slot | Name | Permission | Lambda Role |
|------|------|-----------|-------------|
| 0 | Boot.NS | RW | Kernel meta-variable: namespace root memory object |
| 1 | Boot.Thread | — | Kernel meta-variable: boot thread execution context (no DEMO_CLIST permissions; accessed via CHANGE, not LOAD) |
| 2 | Startup.Config | E | Kernel meta-variable: boot configuration abstraction |
| 3 | Boot.Abstr | E | Kernel meta-variable: boot code abstraction (CR14 source). Note: the simulator's abstraction catalog labels this slot "LED flash" — an alias from early development; the architectural role is Boot.Abstr throughout. |
| 4 | Salvation | E | First user-accessible free variable: security bootstrap |
| 5 | Navana | E | Free variable: namespace master controller |
| 6 | Mint | E | Free variable: capability minter |
| 7 | Memory | E | Free variable: memory allocator |
| 8 | LED[0] | RW | Free variable: LED device 0 (Abstract GT) |
| 9 | LED[1] | RW | Free variable: LED device 1 (Abstract GT) |
| 10 | LED[2] | RW | Free variable: LED device 2 (Abstract GT) |
| 11 | LED[3] | RW | Free variable: LED device 3 (Abstract GT) |
| 12 | LED[4] | RW | Free variable: LED device 4 (Abstract GT) |
| 13 | LED[5] | RW | Free variable: LED device 5 (Abstract GT) |
| 14 | UART | RW | Free variable: serial device (Abstract GT) |
| 15 | Button | R | Free variable: button device (Abstract GT) |
| 16 | SlideRule | E | Free variable: math library abstraction |
| 17 | Timer | RW | Free variable: timer device (Abstract GT) |

**Slots 0–3 are kernel-bound variables.** They are meta-level names installed by the hardware boot sequence itself — the programmer cannot rebind them without hardware privilege. In lambda terms, they are implicitly quantified over the entire boot program; they are not free in the user sense.

**Salvation at slot 4 is the first free variable** in the user sense. From the boot program's perspective, `Salvation`, `Navana`, `Mint`, `Memory`, and the device slots are names available in the initial Γ that the boot code may resolve and apply freely.

The layout above reflects the default foundational boot image (as generated by `server/boot_image.py` and mirrored in the simulator's fallback init path). A custom boot image produced by the Boot Image Designer may reorder or extend slots beyond 17; the 18-slot DEMO_CLIST is the canonical baseline from which all tutorial programs are written.

## 3. Variable Lookup — LOAD / LOADNE / LOADEQ

In lambda calculus, evaluating a variable `x` in environment Γ means looking up the binding Γ(x) and returning its value. On the Church Machine:

```
LOAD CR0, CR6, 4       ; CR0 ← Γ(4)  — resolve variable "Salvation"
```

`LOAD CRd, CRs, #row` dereferences the C-List pointed to by CRs at word offset `row`, and writes the validated Golden Token into CRd. CR6 is the current environment register; `row` is the variable name (an integer index). The mLoad pipeline validates the GT before it reaches CRd — a malformed or revoked binding is rejected at lookup time, not at use time.

### Conditional Variants — Guarded Lookups

`LOADEQ` and `LOADNE` are **type-checked variable lookups**. They carry an ARM-style condition code suffix; the hardware executes or skips the instruction depending on the current condition flags.

```
TPERM CR0, E           ; Is Γ(0) of function type?  Sets Z=1 (yes) or Z=0 (no)
LOADEQ CR1, CR6, 5     ; Lookup Γ(5) ONLY if Z=1 (type check passed)
LOADNE CR2, CR6, 7     ; Lookup Γ(7) ONLY if Z=0 (type check failed — fallback)
```

In a typed lambda calculus, a lookup Γ ⊢ x : T only proceeds if x is bound to a value of type T. The TPERM + LOADEQ pair is the hardware realisation of this: TPERM interrogates the type of a previously resolved variable; LOADEQ performs a second lookup only if the type assertion was satisfied. The lookup is guarded by the prior type witness.

This gives the Church Machine a branchless try-catch discipline: the happy path (type matches) and the error path (type mismatch) are encoded as EQ and NE suffixes on successive instructions, with no branch instructions and no overhead on the happy path.

## 4. Abstractions — Golden Tokens and Closed Lumps

A lambda abstraction `λx.e` has two components: the **binder** `x` (which names the function and makes it referenceable) and the **body** `e` (the computation to perform). The body is a **closed term**: it refers only to names that are either bound in the abstraction's own environment or to the argument `x`. It must not contain free variables that are absent from the abstraction's scope.

On the Church Machine, every abstraction is a **lump** — a power-of-2–aligned binary object in physical memory. The mapping is:

| Lambda component | Hardware equivalent |
|---|---|
| Binder `λx` | E-GT (Enter Golden Token) issued by Mint for this lump's NS slot |
| Body `e` | Lump code section (words 1..cw); CR14 points here during execution |
| Closed environment | Lump C-List tail (words lumpSize-cc..lumpSize-1); CR6 points here during execution |

### The Closed-Term Property

A lump's body accesses capabilities exclusively from its own C-List (the lump-private environment Γ′). It never reaches the caller's C-List. This is the hardware enforcement of the closed-term property: a lambda body may only reference names that are bound in its own binder environment. The lump split that CALL performs (deriving CR6 from the lump's C-List tail, not from the caller's CR6) is the mechanism.

If a lump's body attempted to read the caller's capabilities, it would need a GT pointing to the caller's C-List region — but Mint never issues one. The callee's own CR6 (E-only, M-elevated) points exclusively to the callee's lump C-List tail; the caller's C-List region is simply not addressable from any GT the callee holds. The inaccessibility is structural, not merely a permission restriction.

### E-Only Permission as Function Type

A GT with only `E` permission names a **callable abstraction** — a value of function type. It cannot be used as a data pointer (no R or W), cannot be read as a C-List (no L), and cannot be written into another C-List without the B bit. It can only be applied: `CALL CRn` consumes the E-GT to enter the lump.

In typed lambda calculus terms, the E bit is the function-type tag. Holding an E-GT proves you hold a value of type `T → T`.

### The B Bit — Partial Application

The B (Bind) bit on a GT enables it to be delegated to another abstraction's C-List via `SAVE`. A GT with B=1 can be written into a callee's environment; a GT with B=0 cannot. This corresponds to closing over a variable: when code constructs a new abstraction that captures a capability from its caller, the capability must have B=1 so that `SAVE` can bind it into the new abstraction's C-List. CALL automatically clears B on all preserved registers passed to the callee, preventing unintended delegation of the caller's entire environment.

## 5. Application — CALL and β-Reduction

β-reduction is the rule `(λx.e)[Γ] applied to v ⟹ e[Γ, x ↦ v]`: replace the formal parameter with the argument and evaluate the body in the extended environment.

On the Church Machine, `CALL CRn, #method` performs β-reduction in five hardware steps:

**Step 1 — Validate the function type (`gt_type` + E-permission check)**

For the standard abstraction β-reduction path (the path this document focuses on), the GT in CRn must be an Inform GT (`gt_type = 01`) with E permission set. This is the hardware assertion "CRn holds a value of function type." A NULL GT, a data GT, or an E-GT whose `gt_seq` has been revoked all FAULT here. The check is the hardware equivalent of the typing judgment `Γ ⊢ f : T → T`.

**Step 2 — ChurchNSGate validates the binding (gt_seq + integrity32)**

The GT's `gt_seq` field is compared against the NS entry's `gt_seq`. A mismatch means the capability has been revoked — the binding in Γ that named this abstraction no longer holds. The integrity32 check over NS Word 0 and Word 1 ensures the NS entry itself has not been tampered with. A failing check FAULT. This is the hardware equivalent of ensuring the substituted value is still a valid member of the function space.

**Step 3 — cLoad reads the lump header (closed environment extraction)**

The hardware reads the lump header at `raw_base` to extract `cc` (C-List slot count) and `n_minus_6` (size exponent). From these two fields it computes the lump split:

```
lumpSize = 2^(n_minus_6 + 6)
CR14 (code):   location = raw_base,                   limit = (lumpSize - cc) - 1,  perms = RX
CR6  (c-list): location = raw_base + (lumpSize-cc)×4, limit = cc - 1,              perms = E-only
```

CR6 now points to the callee's private C-List — the closed environment Γ′. This is the substitution: Γ is replaced by Γ′, which is the environment bound into the abstraction at compile time. CR6 carries E-only permission; LOAD from CR6 is M-elevated (the hardware bypasses the L-permission check on CR6 specifically, because CALL already validated E). This means the callee can freely resolve variables from its own C-List via LOAD despite CR6 not holding L permission.

**Step 4 — Push call frame (continuation)**

The caller's E-GT (Word 0) and the next instruction address plus machine indicators (Word 1) are pushed onto the call stack. This preserves the caller's continuation — the "outer λ context" — so RETURN can restore it.

**Step 5 — PC set to method entry (body evaluation begins)**

The hardware reads the method table entry at `memory[raw_base + method_index × 4]` and sets PC to that offset. Execution of the body `e` begins in the callee's closed environment Γ′.

### ELOADCALL — Fused Lookup + Application

```
ELOADCALL CRd, CRs, #row [, #method]
```

ELOADCALL fuses `LOAD + CALL` into a single atomic instruction: resolve variable `row` from environment CRs into CRd, then immediately apply it. There is no intermediate state between the lookup and the application — no window in which another thread could invalidate CRd. In lambda terms, ELOADCALL is `(λx.e)[Γ](Γ(row))` performed atomically.

### RETURN — Restoring the Outer λ Context

RETURN is the dual of CALL. After body `e` evaluates, control must return to the expression that applied the abstraction — the outer λ context. RETURN:

1. Pops the 2-word call frame.
2. Revalidates the caller's E-GT (Word 0) via ChurchNSGate — ensures the caller's NS entry is still live.
3. Re-runs the lump split on the caller's lump header to re-derive CR6 and CR14.
4. Restores PC from the saved NIA (Word 1).

The re-derivation of CR6 from the caller's lump header is critical: it restores Γ (the caller's environment) after Γ′ (the callee's environment) was in force during the call. The machine returns to exactly the lambda context that existed before the application.

## 6. C-List Exchange During CALL

The substitution `(λx.e)[Γ] applied to GT ⟹ e[Γ′]` is implemented as a two-register exchange on CALL and a symmetric restoration on RETURN.

```
Before CALL:
    CR6  → caller's C-List (Boot environment Γ, 18 slots)
    CR14 → caller's code region

CALL CRn executes:
    push { caller's E-GT, NIA }        ; save continuation
    CR6  ← callee's C-List (Γ′)        ; substitute closed environment
    CR14 ← callee's code region        ; enter body e
    PC   ← method entry offset

During callee execution:
    CR6  → callee's C-List (Γ′)        ; body e sees only its own names
    CR14 → callee's code                ; body e runs in lump-private code region

RETURN executes:
    pop { caller's E-GT, NIA }         ; retrieve continuation
    CR6  ← re-derive from caller's lump header (restore Γ)
    CR14 ← re-derive from caller's lump header
    PC   ← NIA                         ; resume outer context
```

**Substitution diagram:**

```
(λx.e)[Γ]  applied to  GT
         ↓ CALL
    e  [Γ′]                 ; Γ′ is lump's private C-List, built at compile time
         ↓ RETURN
    outer context [Γ]       ; Γ restored from call frame
```

No capability in Γ leaks into Γ′ and no capability in Γ′ leaks back into Γ, except through deliberate delegation (SAVE with B=1). The hardware enforces this boundary by making CR6 point to physically disjoint memory regions before and after CALL.

## 7. Environment Layering — Boot → Salvation → Navana

The Church Machine has three nested lambda environments, each a strict subset of the one above it:

### Level 1 — Boot Environment (Γ₀)

The DEMO_CLIST: 18 slots, populated at synthesis time, loaded into CR6 at boot. This is the widest environment — it names all kernel objects and all device abstractions. Only the boot abstraction runs in this environment. Its sole purpose is to call Salvation.

```
Γ₀ = { Boot.NS, Boot.Thread, Startup.Config, Boot.Abstr,
        Salvation, Navana, Mint, Memory,
        LED[0..5], UART, Button, SlideRule, Timer }
```

### Level 2 — Salvation's Environment (Γ₁)

Salvation is a single-purpose security bootstrap abstraction. Its private C-List is a minimal subset of Γ₀ — just the capabilities needed to verify the security pipeline (LOAD, TPERM, LAMBDA correctness). Salvation does not hold the full boot environment. After proving the pipeline works, it calls Navana.

```
Γ₁ ⊂ Γ₀   (strict subset — fewer names, tighter scope)
```

### Level 3 — Navana's Environment (Γ₂)

Navana is the permanent namespace controller. Its private C-List includes all abstractions it manages plus the privileged NS write capability. Navana never RETURNs — it becomes the permanent runtime environment. Every new abstraction Navana installs adds a name to the system, but that name appears in Navana's scope, not in the original Γ₀.

```
Γ₂ ⊂ Γ₀   (different subset — runtime management names)
```

**Principle of Least Authority as environment shrinkage:** each level in the chain sees a smaller, more tightly scoped set of names. The boot environment is intentionally the widest possible — a necessary evil of the cold-start problem. As control passes to Salvation and then to Navana, scope shrinks. User abstractions installed by Navana see only the capabilities Navana explicitly grants them, which are a small subset of Γ₂. The lambda calculus framing makes this hierarchy precise: each level is a distinct environment, and no term can reference a name outside its own environment without an explicit capability binding.

## 8. Domain Purity as Closed-Term Enforcement

The Church Machine's domain-purity rule states: capabilities (Golden Tokens) must never appear as values in the Turing data domain, and raw data values must never appear in C-List slots.

This rule is the hardware expression of **well-scopedness** in lambda calculus. A well-scoped term has no free variables: every name in a term body is bound in the term's enclosing abstraction. A GT appearing in a code word (the Turing domain) would be a capability acting as raw data — a name with no lambda binder in the correct domain. The hardware rejects this because C-List slots are accessed only via LOAD (Church domain), and code words are accessed only via instruction fetch (Turing domain). There is no instruction that reads a code word and treats it as a GT.

Conversely, a raw integer in a C-List slot would be a data value occupying a name slot — a value with no GT binder. Mint validates every C-List slot as a well-formed GT word before the lump is installed. A non-GT value in a C-List slot is a malformed lump; Mint rejects it.

The formal correspondence:

| Lambda property | Church Machine enforcement |
|---|---|
| Well-scoped term: no free variables in body | C-List slots only reachable via LOAD; code words only reachable via instruction fetch |
| Variable must be bound in enclosing λ | GT must appear in the lump's own C-List (installed by Mint) or be delegated in via SAVE with B=1 |
| No raw address as a name | No raw word may occupy a C-List slot — Mint rejects non-GT values in step 8 |
| No name as a raw value | No GT may appear in the Turing data stream — mLoad rejects GT-shaped words in R/W data objects |

Domain purity is not merely a software convention. It is enforced by the hardware's physical separation of the Church domain (L/S/E instructions, CR registers, C-List memory region) from the Turing domain (R/W/X instructions, DR registers, data memory region).

## 9. TPERM as Type-Checking

In the simply-typed lambda calculus, the judgment `Γ ⊢ f : T → T` asserts that in environment Γ, the term `f` has function type. Before applying `f` to an argument, a type checker verifies this judgment.

On the Church Machine, `TPERM CRs, E` is the runtime witness of this judgment:

```
TPERM CR0, E           ; Assert: Γ(CR0) has type "function" (E permission present)
                        ; Sets Z=1 (yes) or Z=0 (no) — no fault on failure
```

The E permission bit is the **function-type tag**. A GT with E set names a callable abstraction — a term of function type. TPERM checks three conditions simultaneously in one cycle:

1. **Permission:** Does the GT in CRs have the E bit set?
2. **Validity:** Does the GT pass gt_seq and integrity32 validation?
3. **Bounds:** Is the specified offset within the GT's region? (Used with the `offset` form.)

If all three pass, Z=1 — the type judgment holds. If any fail, Z=0 — the term is not of function type (or the binding has been revoked).

TPERM does not FAULT on failure. It sets a flag. The code following TPERM uses condition suffixes to either proceed (EQ → type matched) or handle the error (NE → type mismatch). This is the hardware realisation of a branchless type dispatch:

```
TPERM CR0, E           ; Γ ⊢ CR0 : T → T ?
CALLEQ CR0, #method    ; Apply if yes  (β-reduce)
BRANCHNE error         ; Handle error if no
```

The TPERM + conditional CALL sequence is a guarded application: apply only if the type assertion holds. This mirrors the operational semantics of a simply-typed language where application is only defined when the left-hand term is of function type.

### Extended Type Assertions

TPERM covers all six permission types through its preset table:

| TPERM Preset | Lambda Interpretation |
|---|---|
| `E` | Function type — term is callable via CALL |
| `L` | Environment type — term names a C-List (readable namespace) |
| `S` | Sink type — term names a writable C-List |
| `R` | Data type — term names readable data |
| `W` | Mutable data type — term names writable data |
| `X` | Code type — term names executable code (for LAMBDA) |
| `RW` | Mutable data — both read and write |
| `RWX` | Full data type — data and code together |

The `EB` variant (`TPERM CRs, EB`) checks E permission and simultaneously clears the B bit — type-check and permission attenuation in one instruction. This is the hardware equivalent of "assert function type and immediately restrict it to non-delegatable."

## 10. Assembly Example Walkthrough — GC Test (SystemPatterns Section 3)

The following annotated excerpt is the GC-test section of the `system_patterns` assembly example. It exercises the complete lambda sequence: resolve five variables from Γ, assert their types, then β-reduce via CALL.

```asm
; ── Section 3: GC Test (PP250) ────────────────────────────────────────
; Boot C-List in CR6 is the initial environment Γ₀.
; Slots: [4]=Salvation(E) [5]=Navana(E) [6]=Mint(E) [7]=Memory(E) [8]=LED(RW)

LOAD CR0, CR6, 4       ; CR0 ← Γ₀(4)   — resolve variable "Salvation"
                        ; λ equivalent:  x₄ where Γ₀(x₄) = Salvation

LOAD CR1, CR6, 5       ; CR1 ← Γ₀(5)   — resolve variable "Navana"
                        ; λ equivalent:  x₅ where Γ₀(x₅) = Navana

LOAD CR2, CR6, 6       ; CR2 ← Γ₀(6)   — resolve variable "Mint"
                        ; λ equivalent:  x₆ where Γ₀(x₆) = Mint

LOAD CR3, CR6, 7       ; CR3 ← Γ₀(7)   — resolve variable "Memory"
                        ; λ equivalent:  x₇ where Γ₀(x₇) = Memory

LOAD CR4, CR6, 8       ; CR4 ← Γ₀(8)   — resolve variable "LED[0]"
                        ; λ equivalent:  x₈ where Γ₀(x₈) = LED (Abstract GT)

; — Type assertions (Γ ⊢ fᵢ : Tᵢ) ─────────────────────────────────────

TPERM CR0, E           ; ⊢ Salvation : T → T     — function type assertion
                        ; Z=1: E bit present, gt_seq valid, integrity32 intact
                        ; If Z=0: Salvation's binding has been revoked — fatal

TPERM CR1, E           ; ⊢ Navana    : T → T     — function type assertion

TPERM CR2, E           ; ⊢ Mint      : T → T     — function type assertion

TPERM CR3, E           ; ⊢ Memory    : T → T     — function type assertion

TPERM CR4, RW          ; ⊢ LED       : Data_RW   — data type assertion
                        ; LED is not callable; its type is mutable data (RW Abstract GT)

; — β-reduction ────────────────────────────────────────────────────────

CALL CR0, 0xF          ; Apply Salvation, direct method index 15
                        ; λ equivalent:  (λx.e_salvation)[Γ₁] applied to implicit args
                        ;
                        ; imm15=0xF=15: hardware reads memory[raw_base + 15×4]
                        ; as the word-offset entry point (direct dispatch, NOT DR3 escape).
                        ; DR3 escape only applies to c-list-indexed form CALL 15, CRs, #slot.
                        ;
                        ; Hardware steps:
                        ;   1. Validate E permission on CR0 (Salvation GT)
                        ;   2. ChurchNSGate: gt_seq match + integrity32 on NS[4]
                        ;   3. cLoad lump header at Salvation.raw_base:
                        ;         extract cc, n_minus_6 → compute lump split
                        ;   4. CR6 ← Salvation's C-List (Γ₁, E-only) — substitution
                        ;      CR14 ← Salvation's code region (RX)
                        ;   5. Push { caller's E-GT, NIA=next instruction }
                        ;   6. PC ← memory[raw_base + 15×4]  (method table entry 15)
                        ;
                        ; During Salvation: CR6 = Γ₁ (Salvation's closed environment)
                        ;                  CR0..CR4 from caller are visible (inherited)
                        ;
                        ; On RETURN: CR6 restored to Γ₀ (boot C-List re-derived
                        ;            from caller's lump header stored in call frame)

HALT
```

**What this walkthrough demonstrates:**

- Five `LOAD` instructions are five variable lookups in environment Γ₀. Each resolves a name (an integer slot index) to a Golden Token value.
- Five `TPERM` instructions are five type assertions. Together they verify that all five resolved values have the expected types before any application occurs.
- The single `CALL` is one β-reduction step. The hardware performs the environment substitution (Γ₀ → Γ₁), evaluates the body, and on RETURN restores Γ₀.
- The GC-test name reflects the fact that each LOAD triggers a G-bit reset in the NS entry — the hardware's proof that the namespace slot is reachable. This is the "mark" phase of the PP250 GC cycle: accessing a live capability proves liveness.

## Summary

| Concept | Lambda Calculus | Church Machine |
|---|---|---|
| Environment Γ | Map from names to values | C-List (array of GTs indexed by slot number) |
| Variable lookup | Γ(x) | `LOAD CRd, CR6, #slot` |
| Guarded lookup | Type-conditioned Γ(x) | `TPERM` + `LOADEQ` / `LOADNE` |
| Abstraction λx.e | Binder + closed body | E-GT (binder) + lump binary (body + Γ′) |
| Closed term | No free variables outside binder | Lump body accesses only its own C-List |
| Type tag (function type) | `f : T → T` | E permission bit on GT |
| Type judgment | `Γ ⊢ f : T → T` | `TPERM CRs, E` — sets Z=1 if judgment holds |
| Application (f x) | β-reduction | `CALL CRn` — five-step hardware reduction |
| Fused lookup + application | (Γ(x))(arg) | `ELOADCALL CRd, CRs, #row` (atomic) |
| Code application in scope | LAMBDA term (no env switch) | `LAMBDA CRd` — apply X-GT in current scope |
| Substitution (Γ → Γ′) | Environment extension on β | CR6 switch from caller's C-List to callee's C-List |
| Continuation (restore Γ) | Outer λ context after β | `RETURN` re-derives CR6 from saved call frame |
| Well-scopedness | No free variables in body | Domain purity — capabilities and data in disjoint memory regions |
| Partial application (bind) | Capture variable for later | B-bit on GT + `SAVE` into new C-List |
| Environment shrinkage | Smaller Γ = fewer names | Boot Γ₀ → Salvation Γ₁ → Navana Γ₂ (principle of least authority) |
