Church Capability Extensions


                    
Church instructions use 4 dedicated opcodes. LOAD, CALL, and SAVE use a J-type format with 15-bit immediate index and 3-bit CR fields. RETURN, CHANGE, and SWITCH share the custom-0 opcode with R-type format. Hover over each field for details.

MAC Seal Validation

                    

                        
                            

                                Field
                                Bits
                                Description
                            

                        
                        
                            
Version[31:25]7-bit version tag. Bumped on GC sweep to invalidate stale tokens.

                            
Seal[24:0]25-bit FNV-style hash of Location and Limit. Hardware-enforced integrity check.

                        
                    

                    

                        MAC validation is checked on every LOAD and CALL operation. If the stored seal does not match the computed hash of the namespace entry's Location and Limit fields, the operation FAULTs. This prevents tampering with namespace entries and ensures Golden Token integrity.
                    

                    

                        SAVE recomputes the seal when writing to the namespace, preserving the existing version number.
                    


                    
Deterministic Garbage Collection (G-Bit)

                    

                        
                            

                                Phase
                                Action
                                Description
                            

                        
                        
                            
1. MarkSet G-flag on all entriesMarks every non-empty namespace entry as potentially garbage.

                            
2. ScanClear G-flag on reachable entriesWalks CR0-CR15 and call stack. Clears G-flag on entries reachable via valid GTs.

                            
3. SweepCollect still-marked entriesEntries still marked with G-flag are garbage. Zeroed out and version bumped to invalidate stale tokens.

                        
                    

                    

                        The G-bit (tracked per namespace entry, not in the GT permission bits) enables deterministic garbage collection. The three-phase Mark-Scan-Sweep cycle can be run manually via the Dashboard buttons (Mark, Scan, Sweep, or full GC Cycle). Swept entries have their version number incremented, which causes any existing Golden Tokens referencing those entries to fail validation — preventing use-after-free.
