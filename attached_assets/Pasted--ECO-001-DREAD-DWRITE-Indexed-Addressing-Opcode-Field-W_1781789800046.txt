# ECO-001 — DREAD/DWRITE Indexed Addressing + Opcode Field Width Documentation Correction

**Status:** Draft for architect review
**Date:** 2026-06-18
**Author:** Drafted by Claude at K. Hamer-Hodges' direction; pending review and modification by the architect before adoption
**Silicon impact:** Amaranth/Verilog not yet frozen — this ECO targets the next bitstream build, not a shipped product

---

## Summary

Two changes, bundled because the second was discovered while specifying the bit
encoding for the first and must be resolved before any encoding work proceeds:

1. **Add register-offset (indexed) addressing to DREAD and DWRITE.** PP250 supported
   this; its absence from the Church Machine is an oversight, confirmed directly by
   the PP250's original designer, not a deliberate security-motivated omission.
2. **Correct a documentation defect in `isa_reference.md` §1** that misstates the
   instruction word's opcode field as 4 bits with a trailing alignment bit, when the
   per-instruction sections and two independently-decoded worked examples confirm
   the real, hardware-matching format is a 5-bit opcode with no alignment bit.

---

## Part A — Documentation Correction (resolve first; Part B's bit budget depends on it)

### The defect

`isa_reference.md` §1 states the instruction word format as:

```
opcode(4b) | cond(4b) | fld_a(4b) | fld_b(4b) | imm15(15b) | bit0(1b, always 0)
```

with prose claiming "opcode (bits 31:28)... selects one of the 20 instructions
(0–19). Values 20–31 are reserved and fault" and "Bit 0 is always 0... a 1 here
is a FAULT."

Every individual instruction section in the same document (CALL, TPERM, ELOADCALL,
etc.) instead documents `op[4:0]=0x0N` — a 5-bit opcode — with no mention of a
trailing alignment bit anywhere in any per-instruction encoding line.

### Verification performed

Both forms cannot be correct simultaneously: 4+4+4+4+15+1 = 32 bits (the §1 claim,
exactly filling the word with no slack), while 5+4+4+4+15 = 32 bits (the
per-instruction claim, also exactly filling the word, but with a structurally
different field layout and no alignment bit at all).

Two of the document's own worked examples were decoded by hand against each
candidate layout:

**TPERM example** (`isa_reference.md`, TPERM section): hex `0x37080018`, stated to
encode `opcode=6, cond=14, fld_a=1, fld_b=0, imm=0x18`.

- Under the §1 (4-bit opcode) layout: decodes to opcode=3, cond=7, fld_a=0,
  fld_b=8, imm=0xC — **matches none of the stated fields.**
- Under the 5-bit-opcode, no-alignment-bit layout (opcode[31:27], cond[26:23],
  fld_a[22:19], fld_b[18:15], imm15[14:0]): decodes to opcode=6, cond=14, fld_a=1,
  fld_b=0, imm=0x18 — **exact match on every field.**

**ELOADCALL example** (`isa_reference.md`, ELOADCALL section): hex `0x43030002`,
stated to encode `opcode=8, cond=14, fld_a=0, fld_b=6, imm=0x0002`.

- Under the 5-bit layout: decodes to opcode=8, cond=6, fld_a=0, fld_b=6, imm=0x2 —
  matches on opcode, fld_a, fld_b, and imm; **does not match on cond** (decodes to
  6, stated as 14).

### Conclusion and residual uncertainty

The 5-bit-opcode, no-trailing-bit layout is the verified, hardware-matching format
— confirmed exactly by one full worked example and on 3 of 4 fields by a second,
independent worked example. §1's 4-bit-opcode claim, the "values 20–31 reserved"
language, and the "bit 0 always 0, FAULT otherwise" rule should all be treated as
**documentation defects to correct**, not architectural facts to design against.

The single cond-field mismatch in the ELOADCALL example is flagged honestly rather
than resolved here — it may be a transcription error in that one worked example's
hand-annotated comment (as opposed to the hex value itself, which is authoritative)
rather than a deeper problem, but this has not been independently confirmed and
should not be treated as settled.

### Required documentation fix

`isa_reference.md` §1 should be rewritten to state:

```
opcode(5b) | cond(4b) | fld_a(4b) | fld_b(4b) | imm15(15b)
```

with opcode at bits [31:27], no trailing alignment bit, and the "20 instructions,
0–19, values 20–31 reserved and fault" language corrected to reflect that opcode
values 20–31 are simply unused, not specified as fault-triggering anywhere in any
per-instruction section. The HALT/NOP section's framing of bit 0 as a sentinel
should be re-examined against this same correction, since it was built on the same
incorrect premise.

**Practical consequence:** opcode space is 32 values wide (0–31), not 16. 20 are
currently assigned; 12 remain genuinely free for future instructions, not the
"essentially full" picture the §1 defect implies.

## Part B — DREAD/DWRITE Indexed Addressing

### Motivation

DREAD and DWRITE currently take only a compile-time immediate offset
(`#offset`, baked into the instruction word). There is no addressing mode in
which a runtime-computed value (a loop counter, a row index) selects which word
is read or written. This was identified directly while specifying
`EventRouter`'s table-scan logic (see companion abstraction draft,
`event_router_v1.cloomc`): a 16-row linear search over a fixed-width table
required 16 separately-encoded, fully-unrolled DREAD instructions, one per
possible row, because no addressing mode could express "read at base-offset
plus whatever is in this register."

The PP250 supported register-offset (indexed) addressing. Its absence here has
been confirmed by the PP250's original designer to be an oversight in the
Church Machine's instruction set, not a deliberate omission for security
reasons.

### Why a narrow extension, not general computed addressing

A fully general "arbitrary runtime address" addressing mode was considered and
rejected as part of this proposal. Every existing memory-touching instruction in
this ISA validates bounds against a value resolvable at decode time, before the
access occurs — this is a load-bearing property of the capability model, not
an incidental implementation detail. A genuinely free-form computed address
would make bounds-checking depend on an arbitrary runtime value, undermining
the principle that the GT — not the program — defines what is reachable.

The proposal below preserves that property exactly. The runtime register value
is added to the existing compile-time immediate offset; the combined value is
then bounds-checked by the same logic DREAD/DWRITE already use today. The
capability still defines the ceiling. Only the choice of which word beneath
that ceiling is accessed becomes runtime-decidable.

This also reuses an existing precedent rather than inventing a new bit-pattern
convention: IADD already solves exactly this register-vs-immediate problem in
this ISA, using `imm15[14]` as a mode-select bit. This proposal copies that
pattern exactly.

### Encoding (revised for the corrected 5-bit-opcode, 15-bit-imm15 format)

**DREAD — Data Read (opcode 10, 0x0A)**

```
Syntax:  DREAD DRd, CRs, #offset              (immediate form — unchanged)
         DREAD DRd, CRs, #offset, DRx          (indexed form — NEW)
Encoding (immediate): opcode[31:27]=0x0A | cond[26:23] | DRd[22:19] | CRs[18:15] | 1[14] | offset[13:0]
Encoding (indexed):   opcode[31:27]=0x0A | cond[26:23] | DRd[22:19] | CRs[18:15] | 0[14] | base_offset[13:4] | DRx[3:0]
```

- `imm15[14] = 1` → immediate mode, identical to current behaviour. Offset range
  narrows from 15 bits (0–32767) to 14 bits (0–16383 words) — still far larger
  than any lump can be, since `lumpSize` per `CM_LUMP_SPECIFICATION.md` tops out
  at 16384 words.
- `imm15[14] = 0` → indexed mode. `imm15[13:4]` = a 10-bit compile-time base
  offset (0–1023 words). `imm15[3:0]` = DRx, the data register supplying the
  runtime index. Effective offset = `base_offset + DRx`.

**DWRITE — Data Write (opcode 11, 0x0B)** — identical structure, mirrored for
the write direction; same field layout, same mode bit.

### Semantics (indexed mode)

1. `effective_offset = base_offset + DRx`, computed as a 32-bit unsigned add,
   before any permission or bounds check runs.
2. mLoad validates exactly what it validates today: permission (R for DREAD, W
   for DWRITE; CR14 may substitute X for R on DREAD as today), version, CRC
   seal, and that `lump_base + effective_offset` lies within the GT's bounds.
3. BOUNDS fault fires on the *combined* value, exactly as it does today on the
   immediate value. A runtime index walking off the end of a lump faults the
   same way an out-of-range immediate would.
4. `DRx` is read-only input; never written or clobbered by this instruction.
5. All other DREAD/DWRITE semantics (Abstract GT MMIO intercept, DR0
   hardwired-zero rule, DWRITE's thread-sync mirroring) are unchanged and apply
   identically in indexed mode.

**Flags:** N — Z — C — V (no flag writes; unchanged from current behaviour).

**Faults:** identical fault table to current DREAD/DWRITE. No new fault codes.

### Verified impact on existing code

Every DREAD/DWRITE immediate offset across all 17 real `.cloomc` example files
in `examples_extracted/` was scanned directly. The largest value used anywhere
is 105 (`ada_note_g.cloomc`); the overwhelming majority are single-digit
LED/register offsets (`led_dr_test.cloomc`, `perm_attack.cloomc`,
`petname_demo.cloomc`). None approaches the proposed 14-bit ceiling (16383).
**This change is purely additive against the existing corpus — no existing
source requires re-assembly.**

### Worked example — what this enables

`EventRouter`'s 16-row event table (`event_router_v1.cloomc`), today requiring
16 hand-unrolled fixed-offset DREAD instructions, becomes:

```
; DR1 = row counter, DR2 = event GT to match, CRheap = table base
loop:
    DREAD DR3, CRheap, #0, DR1      ; word0 of row[DR1] -- indexed read
    MCMP  DR3, DR2
    BRANCH EQ, found
    IADD  DR1, DR1, #2              ; advance to next 2-word row
    MCMP  DR1, #32                  ; 16 rows * 2 words
    BRANCH LT, loop
    ; not found
```

This is not specific to `EventRouter` — it generalizes to any abstraction
holding a heap-resident array of fixed-width records, including `Contact`'s own
pet-name table.

### Explicitly out of scope

- No new opcode; no growth in the documented instruction count beyond 20.
- No change to CALL, ELOADCALL, LOAD, or any capability-traversal instruction —
  scoped entirely to data-object word access via DREAD/DWRITE.
- No change to GT format, permission bits, or the c-list mechanism.
- No change to the bounds-checking *guarantee* — only to when the offset being
  checked is determined (compile time vs. runtime); the check itself is
  unchanged.

### Open items for architect decision

1. **`base_offset` field width (currently proposed at 10 bits, 0–1023 words).**
   This comfortably covers any single Zone in a 16K-word lump, but a sanity
   check against real Zone-size expectations in `CM_LUMP_SPECIFICATION.md` is
   worth doing explicitly rather than treating 10 bits as self-evidently
   correct.
2. **Hardware cost.** This adds one 32-bit adder and one bit of mode-decoding
   to the DREAD/DWRITE datapath in `hardware/*.py` (Amaranth). Given the
   silicon is not yet frozen, this is presented as a content question for
   whoever owns the Amaranth source, not a feasibility question — the change
   appears straightforward, but actual LUT/gate cost on the Ti60 F225 target
   has not been measured here.
3. **Single cond-field mismatch noted in Part A.** Should be independently
   re-verified against the ELOADCALL example's original source (or against
   `hardware/core.py`'s actual decode logic directly) before this ECO is
   considered fully closed, since Part B's bit arithmetic depends on Part A's
   correctness.
